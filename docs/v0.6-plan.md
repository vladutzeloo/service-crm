## 1. Audit summary

### 1.1 Mapping to the domain

ROADMAP §0.6.0 is the **second** ticket slice and adds the
`knowledge/` blueprint. Per
[`architecture-plan.md`](./architecture-plan.md) §3.2 + §4 it owns nine
new tables across two blueprints:

`tickets/` (extension):

- `service_intervention` — the field-job header (start/end times,
  technician, ticket FK).
- `intervention_action` — discrete actions performed.
- `intervention_finding` — diagnoses / observations.
- `part_master` — lightweight catalog (not a warehouse — blueprint §2).
- `service_part_usage` — per-intervention part draws (FK to
  `part_master` is nullable to allow ad-hoc snapshots).

`knowledge/` (new blueprint):

- `checklist_template` + `checklist_template_item` — admin-edited
  recipes.
- `checklist_run` + `checklist_run_item` — frozen snapshots per
  intervention.
- `procedure_document` — searchable Markdown.
- `procedure_tag` + `procedure_document_tag` (M2M) — tagging.

The v0.5 `TicketAttachment.intervention_id` placeholder is finally
wired up — column added in this milestone's first migration with a
nullable FK so existing rows continue to validate.

### 1.2 What's already decided

From [`architecture-plan.md`](./architecture-plan.md) (approved 2026-05-10):

- Model shapes for all nine tables (§4.1).
- Frozen-snapshot rule: `ChecklistRun.snapshot` and
  `ChecklistRunItem.template_item_id` are immutable after creation;
  template edits never mutate historical runs.
- `PartMaster.is_active` soft-delete.
- ULID PKs across the board.
- Procedure search reuses the same dialect-aware FTS pattern as
  clients/equipment/tickets (Postgres GIN expression-index, SQLite
  `LIKE` fallback).
- Shared upload pipeline from 0.5 handles intervention photos.

### 1.3 What v0.6 has to actually decide

1. **Where intervention models live.** New `interventions/` blueprint
   vs. extending `tickets/`. See §3.1.
2. **Action/finding model shape.** Free-text only vs. template-driven.
   See §3.2.
3. **Checklist run materialisation.** Eager (snapshot on run-create)
   vs. lazy (resolve on render). See §3.3.
4. **Procedure body format.** Markdown (rendered server-side) vs. plain
   text vs. limited HTML. See §3.4.
5. **Part usage de-duplication.** Allow the same part twice on one
   intervention (kept rows; sum at display) vs. enforce unique
   `(intervention_id, part_id)`. See §3.5.

## 2. Assumptions

1. **One technician per intervention at any time.** Multi-technician
   interventions are a v1.3 concern (per
   [`docs/blueprint.md`](./blueprint.md) §21 phase 5). v0.6 stores a
   single `technician_user_id`. Reassignment edits the row.
2. **`ServiceIntervention.ticket_id` is required.** No standalone
   interventions; every intervention belongs to a ticket. The
   blueprint says interventions are the technician's "session inside a
   ticket".
3. **Actions and findings are free-text in v0.6.** Templated actions
   (e.g. "replaced spindle bearing") get a lookup table in v0.7
   alongside maintenance. v0.6 stores `description` + an
   `is_root_cause` flag on findings.
4. **A checklist run snapshots its template at create time.** Editing
   the template afterwards never mutates existing runs. Tested via a
   property test that mutates a template after a run, then asserts the
   run's `snapshot` is unchanged.
5. **Procedure body is Markdown stored as text.** Server-side render
   happens with a tiny allowlist (paragraphs, headings, lists, code
   blocks, links) — no raw HTML, no script tags. Plain text is the
   v1.0 storage format; the renderer is best-effort and replaceable.
6. **Procedure search returns ranked results on Postgres**
   (`ts_rank_cd` against the GIN expression-index) and unranked
   substring matches on SQLite. Locale-agnostic (`'simple'` config).
7. **Part usage allows duplicate `(intervention_id, part_id)`.** Real
   field jobs draw the same SKU at different times in a session;
   coalescing happens at display time, not in storage.
8. **Photo uploads on an intervention store under
   `instance/uploads/interventions/<intervention_hex>/`.** Same
   pipeline as ticket attachments — extension + magic-byte check,
   Pillow re-encode to WebP q85, ≤ 2048 px long edge, 25 MB cap.
9. **No technician-side state machine on interventions.** They
   represent recorded work, not workflow. A ticket can have many
   interventions over its lifetime; closure of the intervention is
   marked by setting `ended_at`.
10. **Translation strategy mirrors 0.5.** `tickets/_translations.py`
    already covers status/type/priority; this PR introduces
    `tickets/_intervention_translations.py` for action/finding kind
    labels (only `is_root_cause`'s "Root cause" / "Observation" pair),
    and `knowledge/_translations.py` for checklist item kinds
    (`bool`/`text`/`number`/`choice`) — three short modules total.

## 3. Options considered

### 3.1 Where the intervention models live

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Extend `tickets/`** (architecture-plan §4 mapping) | New models in `tickets/models.py`; new `interventions_services.py` module split for clarity; routes mounted under `/tickets/<hex>/interventions/...` | Matches architecture plan verbatim; interventions are tightly coupled to a ticket so the FK chain is local; no cross-blueprint imports beyond the ones we already have | `tickets/` becomes a fat blueprint (5 → 10 models, ~5 → ~9 k LOC) |
| B — New `interventions/` blueprint | Stand-alone | Slimmer module | Diverges from approved architecture plan; every route still needs the ticket context; the M2M between interventions and parts becomes cross-blueprint |
| C — Mixed: intervention models in `tickets/`, parts in their own `parts/` blueprint | Hybrid | None for v0.6 | Parts are only ever used by interventions; carving them out is premature abstraction |

### 3.2 Action / finding model shape

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Free text + `is_root_cause`** (plan §4.1) | `description: Text`, plus a bool on findings | Matches blueprint verbatim; one schema; ready to layer template-driven actions in v0.7 alongside maintenance recipes | Free text means no clean reporting yet — but reporting lands in v0.8 |
| B — `kind` FK + free text | Lookup + text | Cleaner reporting | Lookup tables for v1 actions/findings deferred to v1.3 (`docs/blueprint.md` §21 phase 5); building the FK in v0.6 wires up a half-empty table |
| C — JSON structured fields | Flex | Reports are not free | Schema-as-data; reporters in v0.8 have to learn the dialect |

### 3.3 Checklist-run materialisation

| Option | When the snapshot freezes | Pros | Cons |
| --- | --- | --- | --- |
| **A — Eager: `ChecklistRun` build copies the template into `snapshot` (JSON) and creates one `ChecklistRunItem` per template item at create time** | At run-create | Run row is self-contained; deleting the template is allowed; matches the constraint "historical runs survive template edits"; ORM round-trips are cheap | One extra write per template-item at run-create; mitigated by `session.bulk_save_objects` (template items are typically ≤ 30) |
| B — Lazy: store `template_id` + `template_version` only; resolve at render time | At render | Smaller writes | Edit-after-snapshot bug returns immediately; needs versioned templates (deferred to v1.x) |
| C — Hybrid: eager for the JSON blob, lazy for items | Mixed | Both | Conceptually muddy; two sources of truth |

### 3.4 Procedure body format

| Option | Storage | Renderer | Pros | Cons |
| --- | --- | --- | --- | --- |
| **A — Markdown text, server-side rendered via a tiny in-package renderer** (regex-based: headings, paragraphs, lists, code, links) | Text | ~80 LOC | No new dep; deterministic; safe (HTML-escapes everything by default) | Limited markdown subset (no tables, no images inline) — table support adds in v0.7 if needed |
| B — Markdown via `mistune` / `markdown-it-py` | Text | Library | Full markdown | New dep; XSS exposure if config drifts |
| C — Plain text | Text | None | Trivial | Loses structure; reviewers asked for at least lists + code blocks |

### 3.5 Part-usage uniqueness

| Option | Constraint | Pros | Cons |
| --- | --- | --- | --- |
| **A — No uniqueness; coalesce at display** | None | Real workflows draw the same part twice (e.g. one before lunch, one after a wrong-part return); preserves audit fidelity | Display has to coalesce by `part_id` |
| B — `UNIQUE(intervention_id, part_id)`; bump `qty` on retry | Constraint | Reads cleanly | Loses the timing detail; conflicts with v0.5 "every state change writes a history row" philosophy |
| C — Same as B but expose a separate `part_usage_event` table | Two tables | Best of both | Premature; reporting lands in v0.8 and can introduce this then if needed |

### 3.6 Slicing (per-PR ordering)

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — One PR for v0.6** | All of interventions + parts + knowledge in a single release-PR | Schema lands in one migration; CHANGELOG gets one entry per milestone; reviewers see the milestone as a unit | ~5 k LOC review |
| B — Two PRs: (1) interventions + parts, (2) knowledge | Two release-PRs in the same milestone | Smaller diffs | Two migrations; second PR has to extend `TicketAttachment` only after the first lands |
| C — Three PRs split by surface | Smaller still | Tiny diffs | Same problem as the v0.5 §3.6 C — synchronisation cost outweighs the LOC win |

## 4. Recommendation

**Pick A on every option** (§3.1A through §3.6A). All A choices match
the approved architecture plan and the patterns shipped by 0.5; the
exceptions in §3.1 and §3.2 each have a v1.3 plan that would invalidate
v0.6 effort if taken now.

## 6. Open questions

Resolved during implementation; recording the defaults here.

1. **Intervention start vs. now.** Default `started_at = clock.now()`
   on create; admin/manager can edit; technician form just shows it.
2. **Ending the intervention.** Setting `ended_at` is a separate
   form action ("Stop intervention") — no auto-end.
3. **Photo aspect ratios.** Re-encode keeps the natural ratio; the
   grid in the template uses CSS `object-fit: cover` for thumbnails.
4. **Markdown table support.** Excluded from v0.6's renderer.
5. **Procedure visibility.** All procedures are visible to every
   authenticated user; per-role visibility is v1.x.
6. **Part code casing.** Stored as-typed; uniqueness check is
   case-insensitive (mirror `Client.email` pattern).
7. **Checklist item kinds.** `bool`, `text`, `number`, `choice` —
   the `choice_options` JSON column is required only for `choice`.
