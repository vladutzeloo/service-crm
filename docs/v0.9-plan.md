> Output of the `/architecture-audit` skill for ROADMAP §0.9.0, drafted
> on branch `claude/upgrade-go-0.9-WU69I`. Follow-on to
> [`v0.7-plan.md`](./v0.7-plan.md); reuses every decision taken in 0.5

## 1. Audit summary

### 1.2 The 0.8 dashboard gap

`VERSION` reads `0.7.0`. ROADMAP §0.8.0 (operational dashboard +
reporting) is **unshipped**: there is no `service_crm/dashboard/`
blueprint, no `templates/dashboard/`, no reports surface. The §1.1
functional-completeness bar **explicitly requires** a manager dashboard
and CSV-exportable reports (see
[`v1-implementation-goals.md`](./v1-implementation-goals.md) §1.1
"Manager review" and "Reports").

This plan does **not** silently skip 0.8. It addresses the gap head-on
in §3.0 below; the recommended path is to absorb a *minimal*
operational-dashboard slice into the 0.9 sequence (W0 — see §4.1) so
the §1.1 bar can be ticked at tag time. The full 0.8 ambition
(every KPI tile, every report) is **not** in scope here; the minimum
necessary to clear §1.1 is.

### 1.3 What 0.9 has to actually decide

1. **How to handle the 0.8 gap.** Three options in §3.0.
2. **One mega-PR vs. a sequence of slices.** Options in §3.1.
3. **Where the observability plumbing lives** (`shared/logging.py` vs.
   inline middleware vs. third-party). See §3.4.

## 2. Assumptions

1. **Feature freeze means feature freeze.** No new entity types, no
   new state machines, no new business rules in 0.9. Anything that
   reads like "while we're in here, let's also add X" goes to the
   1.x backlog, not 0.9. The one exception is W0 (minimum dashboard
   slice) and W7 (GDPR endpoints) — both are §1.1 / §1.8 bars that
   we deferred from earlier milestones, not new scope.
2. **The 100 % branch coverage gate stays on.** Every workstream that
   adds code adds the tests for it; W4's logging middleware is the
   trickiest case (asserting log lines without coupling the assertion
   to formatting cosmetics).
3. **No new third-party services.** Sentry / Datadog / Honeycomb are
   *documented hooks* (W4 §1.5) — we expose `SENTRY_DSN` and a logging
   handler; we do not vendor a client. Same posture as the 0.1.0 ADR
   `0001-flask-vs-fastapi.md` Single-VPS principle.
4. **CI gets a fourth + fifth job.** `pip-audit` runs alongside the
   existing 3.11 × {SQLite, Postgres} matrix; Lighthouse-CI runs on a
   single canonical mobile config (`mobile`, slow-4G simulated).
   axe-core runs as a Playwright pytest plugin against the existing
   Postgres job's app.
5. **The "reference dataset" for perf is seeded with `factory-boy`.**
   No production data ever lives in CI. The seed lives in
   `tests/perf/_seed.py` and is reusable from the docs/operator-runbook
   walkthrough.
6. **All new strings translated.** RO + EN catalogs grow with the
   GDPR endpoints, the `/readyz` payload, the dashboard slice (W0),
   and any newly-surfaced error messages from W3. The W8 catalog
   completeness gate ratchets on the moment it lands; it cannot ship
   in the same PR that introduces the strings it would block.
7. **Manual real-device QA happens on the release candidate, not on
   `main`.** Findings go into the `## [0.9.0]` changelog block as
   shipped fixes; the QA itself is gated by §1.7's release checklist.
8. **The 0.8 catch-up slice is exactly the minimum to clear §1.1, not
   the full 0.8 milestone.** The full 0.8 scope (every KPI tile from
   the architecture plan, every report from the blueprint) gets a
   real `docs/v0.8-plan.md` *after* 1.0 lands, if it lands at all —
   it may be re-prioritised behind 1.1 customer portal.

## 3. Options considered

### 3.0 How to handle the unshipped 0.8 dashboard

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Absorb a minimum dashboard slice into 0.9 as workstream W0** (this plan's recommendation) | One small dashboard blueprint with admin + operator views, drillable KPIs only (no reports surface) | §1.1 can be ticked; 1.0 ships with the manager / technician views the architecture plan promised; smallest possible scope-creep into 0.9 | 0.9 grows by ~1.2 k LOC vs. a "pure hardening" milestone |
| B — Tag 0.8.0 first as a separate release, then start 0.9 | Two milestones, two tags | Cleanest history; 0.8 gets a proper plan PR | Doubles the wall-clock time to 1.0; the full 0.8 scope (every KPI + every report) is far larger than the §1.1 bar actually needs |
| C — Drop the dashboard requirement from §1.1 for 1.0 | Doc edit only | Zero engineering work | §1.1 is the *production-ready* bar; a CRM without a manager dashboard is not production-ready; this is a regression on a locked decision (`docs/v1-implementation-goals.md` §0) |

**Pick A.** The §1.1 functional-completeness bar is non-negotiable
and the minimum dashboard slice is ~1.2 k LOC — small enough to land
inside 0.9 without breaking the milestone's coherence. The 0.8 reports
surface (CSV exports for tickets-by-status, parts-used, etc.) is
folded into W0 at the *minimum* level: one CSV export route per
mandated report in §1.1, sharing one templated download view.

### 3.1 One mega-PR vs. a sequence of slices

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Sequence of small PRs, one per workstream** (this plan's recommendation) | 8 PRs (W0 → W8) merged in order against the same branch | Each PR is reviewable in < 1 hour; bisect-friendly; CI greens incrementally; risk per merge is small | More merge-management overhead; CHANGELOG `## [Unreleased]` grows over time |
| B — One PR for everything | Single PR carrying ~9 k LOC | One CHANGELOG block; single tag-time gate | Unreviewable; CI flake compounds; a single rework in W6 (docs) blocks W3 (security) for no reason |
| C — Two PRs: "core" (W0+W2+W3+W4+W7) and "polish" (W5+W6+W8) | Hybrid | Compromise between A and B | Still ~5 k LOC per PR; the W0 dashboard ends up bundled with security headers which makes the PR description incoherent |

**Pick A.** The 0.7 plan (`v0.7-plan.md` §1.4) explicitly defaulted
to one PR for ~4.6 k LOC of *new feature work*; 0.9 is twice that and
the workstreams have no shared model code. The 0.9 branch
(`claude/upgrade-go-0.9-WU69I`) stays open the whole milestone;
sub-PRs target it; one final merge to `main` happens at tag time.

### 3.2 Workstream sequencing

Within option §3.1A, the recommended order:

```
W0 (dashboard min)  →  W4 (observability)  →  W3 (security)  →
W2 (perf)           →  W7 (GDPR)           →  W5 (operability) →
W8 (polish)         →  W6 (documentation, last)
```

Rationale:

- **W0 first** because §1.1 must be tickable before anything else
  matters; also adds the routes W2 / W8 will benchmark.
- **W4 before W3** because the security hardening PR (W3) wants to
  assert structured log lines (`"unauthorized access denied"`) — the
  JSON logger has to exist first.
- **W3 before W2** because perf testing needs to traverse a logged-in
  user with realistic RBAC restrictions; W3's role decorators land
  first so the perf seed sets up the right actors.
- **W7 (GDPR) before W5 (operability)** because the operator runbook
  (W5 wraps) needs to document the GDPR endpoints' admin UI.
- **W8 (polish) last among code workstreams** because Lighthouse +
  axe-core are CI gates — they must catch *regressions* introduced
  by W0 – W7, so they ratchet on at the end.
- **W6 (documentation) very last** so the docs reflect what actually
  shipped, not what we planned to ship.

### 3.3 Performance fixture scope

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Full §1.3 reference dataset, seeded lazily in a session-scoped fixture** | 10 k clients, 50 k equipment, 100 k tickets, 100 techs, 3 y audit | Matches the v1 bar verbatim; once-built `tests/perf/.cache/` survives across runs | Seed takes ~3 – 5 min the first time; Postgres-only (SQLite can't seed 100 k rows in any reasonable wall-clock) |
| B — Scaled-down "hardening" fixture (1 k / 5 k / 10 k / 10) | Smaller seed | Faster first run; usable on SQLite | The §1.3 P95 budgets don't apply — every offender measured here is "probably fine at 10×" rather than measured |
| C — Synthetic but Postgres-only; SQLite leg of the matrix skips perf | Hybrid | Honest about the SQLite limit | Need a `pytest.mark.perf_postgres` marker + CI logic |

**Pick A**, gated by `pytest -m perf` so the default `pytest` run
stays fast. The CI matrix gets a fourth job that runs only the `perf`
marker against Postgres; the matrix exits non-zero if any P95 budget
in §1.3 is exceeded. The cache directory (`tests/perf/.cache/`) is
git-ignored.

### 3.4 Observability plumbing placement

| Option | Where | Pros | Cons |
| --- | --- | --- | --- |
| **A — `service_crm/shared/logging.py`** with `init_app(app)` style + `RequestContextFilter` that reads from the existing `REQUEST_ID_CTX` / `ACTOR_CTX` | Shared module | Same pattern as `shared/scheduler.py`; one stdlib logger config; tests can monkeypatch the handler list | None worth listing |
| B — Inline middleware in `service_crm/__init__.py` | Inside the factory | One less file | Tangles factory wiring; harder to test in isolation |
| C — Third-party (`structlog`, `loguru`) | New dependency | Free features (renderers, processors) | New dep is overkill — every feature §1.5 wants is in stdlib `logging`; we'd be carrying a dep for cosmetics |

**Pick A.** The factory calls `shared.logging.init_app(app)` after
`init_extensions`. The filter reads from `contextvars` (which the
auth middleware already populates per request). Each request emits
one line via Flask's `after_request` hook; the line is a single JSON
object with `request_id`, `user_id`, `route`, `status`, `latency_ms`,
`length`. App-internal log calls inherit `request_id` automatically
through the filter.

### 3.5 `/readyz` semantics

| Option | Checks | Pros | Cons |
| --- | --- | --- | --- |
| **A — DB reachable + migrations at head + (when SCHEDULER_ENABLED) scheduler running** | Three checks | Matches §1.5 "Readyz failures during boot block container start"; the migrations-at-head check is the one that catches "ops forgot `flask db upgrade`" | The migrations check costs one query — fine, cached for 5 s |
| B — DB reachable only | One check | Faster | Misses the most common ops failure (forgot to upgrade) |
| C — DB + scheduler + uploads dir writable + a synthetic "round-trip" insert | Heavy | Catches everything | A round-trip insert in `/readyz` mutates the audit log; weird operationally |

**Pick A.** The check returns HTTP 200 with a JSON body listing each
sub-check's status; 503 if any sub-check fails. The migrations check
compares Alembic's `current_head()` against `current_rev()` on the DB.
A 5-second in-process cache (`functools.lru_cache` + TTL) keeps the
readyz endpoint cheap under k8s-style probe storms — irrelevant on
the single-VPS target, but free.

### 3.6 `/metrics` shape

| Option | Format | Pros | Cons |
| --- | --- | --- | --- |
| **A — `prometheus_client` Prometheus text format, off by default, behind `METRICS_ENABLED`** | Standard | Standard tool can scrape; off-by-default keeps the surface area zero unless asked | New dep (~lightweight) |
| B — Custom JSON | None | No dep | Nothing scrapes a custom JSON format |
| C — Skip entirely | None | No work | §1.5 requires the endpoint |

**Pick A.** `prometheus_client>=0.20` joins `pyproject.toml`. Default
metrics: HTTP request count, duration histogram, in-flight requests,
audit-events-emitted counter, scheduler-tick counter. Custom metrics
land as their domain code lands — *no* per-blueprint metrics in this
milestone; that's a 1.x scope-creep risk.

### 3.7 GDPR export shape

| Option | Format | Pros | Cons |
| --- | --- | --- | --- |
| **A — Single JSON file** (`{"client": {...}, "contacts": [...], "tickets": [...], ...}`) per customer | JSON | Easy to diff against the source DB; matches the §1.8 "JSON dump of every row referencing that customer" wording verbatim; integrates with the audit log naturally | Large customers → large files (a multi-MB JSON is fine to download but awkward to email) |
| B — ZIP-of-CSVs (one CSV per entity table) | ZIP | Spreadsheet-friendly | Implementation complexity (multiple writers, ordering); some entity references span multiple sheets |
| C — Both, with a query string | Both | Caller picks | Twice the test surface |

**Pick A.** JSON dump matches §1.8 verbatim. Anonymisation rules
documented in `docs/operator-runbook.md`; the export endpoint
returns the raw dump, not a redacted one — redaction is the receiver's
job, and we audit-log every export.

### 3.8 GDPR forget semantics

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Anonymise: scrub personal columns on `Client` / `Contact` / `Location`; leave service history; flip `Client.is_active = False` + new `Client.is_anonymised = True` flag** | Soft, reversible-by-restore | Preserves service history (warranty claims, support obligations) — a v1 *single-tenant* CRM cannot afford to delete service history; matches the §1.8 "anonymises personal fields while preserving service history" wording verbatim | Reversal possible only from backup — but that's by design; not a flaw |
| B — Hard-delete the customer + cascade everything | Permanent | Clean DB | Destroys service history; legally risky for warranty audits; user wouldn't pick this in practice |
| C — Hard-delete personal columns (`UPDATE … SET name = NULL, …`) but preserve FK rows | Almost-A | None vs. A | Same as A in effect; A is clearer about intent |

**Pick A.** The endpoint:

- POST `/admin/gdpr/clients/<hex>/forget` (admin-only, role-gated).
- Sets `Client.name`, `Client.tax_code`, `Contact.email`,
  `Contact.phone`, `Location.address_*` to their anonymised
  sentinels (`"GDPR-forgotten-<short-token>"` / `NULL`).
- Sets `Client.is_active = False` + `Client.is_anonymised = True`.
- Writes one audit row carrying the actor, the request id, and the
  short-token for postmortem reconciliation.
- Idempotent: calling twice on an already-anonymised client returns
  200 with `{"status": "already-forgotten"}`.

### 3.9 Lighthouse / axe-core CI integration

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — `lighthouse-ci` GitHub Action against the `docker compose up`-ed app on a CI-side ephemeral Postgres; axe-core via the `axe-playwright-python` pytest plugin** | Two CI integrations | Each tool's standard recipe; results uploaded as build artefacts | Two new CI tools — but both are free / FOSS and used widely |
| B — Run both inside the pytest suite locally; commit golden reports | Single tool | Self-contained | Tests now require Chrome at runtime; flaky on contributor machines; commits-of-binaries growth |
| C — Skip Lighthouse, keep axe-core | Partial | One fewer tool | §2.6 requires Lighthouse Performance ≥ 90 |

**Pick A.** Lighthouse-CI lives in `.github/workflows/lighthouse.yml`,
triggered on `main` and on PRs labelled `phone-impact`. axe-core runs
under the existing pytest matrix as `tests/a11y/test_axe_smoke.py`;
the test asserts zero `serious` or `critical` violations on the eight
P1 pages enumerated in §1.1.

## 6. Open questions

Resolved during implementation; recording the defaults here so each
sub-slice PR doesn't have to re-litigate them.

1. **Sequence of sub-PR branches.** All target
   `claude/upgrade-go-0.9-WU69I`; each sub-PR is opened against this
   branch, not against `main`. `main` sees one merge at tag time.
2. **Catch-up release tag.** `v0.8.0` is **not** tagged; the
   minimum dashboard slice ships as part of `v0.9.0`. The
   `CHANGELOG.md` `## [0.9.0]` block documents the minimum-dashboard
   work explicitly. Follows the `v0.4.0` catch-up precedent
   (PR #21 "release: 0.4.0 (catch-up, bundles 0.3 + 0.4)").
3. **Lighthouse failure threshold.** Initial run calibrates against
   the just-shipped 0.9 build; the budget *ratchets on* once green.
   First red run blocks merge; first green run pins the gate.
4. **`pip-audit` policy.** CI fails on any `HIGH` or `CRITICAL`
   advisory at PR time. `MEDIUM` advisories are logged but not
   blocking — consistent with the §1.4 wording.
5. **RBAC matrix coverage.** One test per `(role, P1 action)` cell
   where P1 actions are the union of every route mentioned in
   `v1-implementation-goals.md` §1.1. Estimated ~80 cells; the
   `tests/security/test_rbac_matrix.py` driver is one parametrised
   test, not 80 hand-written ones.
6. **GDPR export size cap.** None in v0.9; if a single customer's
   JSON dump exceeds ~50 MB the operator runbook recommends running
   `pg_dump` directly against the per-customer rows instead. A
   streaming export is a 1.x scope.
7. **CHANGELOG style.** Each sub-PR appends to `## [Unreleased]`
   under the appropriate Keep-a-Changelog heading; the tag-time PR
   renames `## [Unreleased]` to `## [0.9.0]` and adds a fresh empty
   `## [Unreleased]` above it (same shape as every prior release).
8. **`/readyz` migration check on SQLite.** Alembic `current_rev()`
   on SQLite uses the same `alembic_version` table; the check works
   unchanged. No SQLite-specific branch.
9. **`SECURITY.md` disclosure email.** Default to the address
   already listed in the GitHub repo `SECURITY` setting (if any);
   otherwise the maintainer's GitHub email per `pyproject.toml`
   `authors[0]`. W3 PR confirms with the maintainer before merge.
10. **ADR formatting.** Use the existing ADR 0001 template
    (Status / Context / Decision / Consequences / References) for
    every new ADR; do not import the MADR or Y-Statement variants.
