## 1. Audit summary

### 1.1 Mapping to the domain

ROADMAP Ā§0.7.0 closes out the v1 data model with the maintenance and
planning slices. Per
[`architecture-plan.md`](./architecture-plan.md) Ā§3.2 + Ā§4 it owns
**seven** new tables across two brand-new blueprints (no schema-extension
this time, both blueprints are new):

`maintenance/` (new blueprint):

- `maintenance_template` ā€” reusable recipe (cadence + estimated time +
  optional `checklist_template` linkage).
- `maintenance_plan` ā€” per-equipment schedule
  (`cadence_days`, `last_done_on`, `next_due_on`).
- `maintenance_task` ā€” generated due-task instance from a plan.
- `maintenance_execution` ā€” completed task record linked back to an
  intervention.

`planning/` (new blueprint):

- `technician` ā€” 1:1 with `user_account` but separate so capacity and
  timezone live off `user_account`.
- `technician_assignment` ā€” ticket / intervention assignment.
- `technician_capacity_slot` ā€” per-day declared capacity.

### 1.2 What's already decided

From [`architecture-plan.md`](./architecture-plan.md) (approved 2026-05-10):

- Model shapes for all seven tables (Ā§4.1).
- `MaintenancePlan.next_due_at` is recomputed by an APScheduler job;
  non-null index for "due soon" queries.
- `MaintenanceTask.is_done` flips only via `MaintenanceExecution`
  insert. A standalone update on `is_done` is a service-layer error.
- `TechnicianAssignment` has a DB-level CHECK so at least one of
  (`ticket_id`, `intervention_id`) is set.
- ULID PKs across the board.
- Soft-delete via `is_active` on `Technician` and `MaintenancePlan`.
- `MaintenanceTemplate` may reference an existing `ChecklistTemplate`;
  the link is optional (a template can ship purely as cadence +
  description without a checklist).

### 1.3 What v0.7 has to actually decide

1. **Where the maintenance models live.** New `maintenance/` blueprint
   (matches architecture plan) vs. extending `equipment/`. See Ā§3.1.
2. **What "due" means.** Store `next_due_on` as `Date` (clear UI, no
   timezone games) vs. `DateTime(timezone=True)` (more precise). See Ā§3.2.
3. **Where the APScheduler instance lives.** New `shared/scheduler.py`
   that the factory wires up, with jobs registered via the blueprint
   `init_app` style vs. inline-in-blueprint. See Ā§3.3.
4. **Task generation window.** Generate a task only for the *next* due
   date vs. for every due date inside a horizon (e.g. 14 days). See Ā§3.4.
5. **Escalating an overdue task to a ticket.** One-click POST that
   creates a `ServiceTicket` linked back to `MaintenanceTask.ticket_id`
   vs. an explicit redirect into the new-ticket form. See Ā§3.5.
6. **Technician roster vs. User.** Separate `technician` row keyed by
   `user_id` (architecture plan) vs. promoting `User.is_technician`.
   See Ā§3.6.

## 2. Assumptions

1. **Cadence is whole days.** `cadence_days = 30` means "due 30 days
   after `last_done_on`". Hourly or runtime-based cadences are v1.3
   (per [`docs/blueprint.md`](./blueprint.md) Ā§21 phase 5).
2. **`MaintenancePlan.next_due_on` is a derived value.** The service
   layer is the only writer; the APScheduler job recomputes it as
   `last_done_on + cadence_days` (or `created_at::date + cadence_days`
   when never executed). A direct mutation is a service-layer error.
3. **One open `MaintenanceTask` per plan.** Generating the next task
   waits until the previous one is marked done. This keeps the dashboard
   counts honest and matches how field crews actually work.
4. **Escalating a task to a ticket** creates a new `ServiceTicket`
   pre-populated with the equipment's client + the maintenance
   template's name as the ticket title, type `preventive`, priority
   `normal`. `MaintenanceTask.ticket_id` carries the FK back.
5. **`Technician.user_id` is unique.** One technician row per user.
   Bumping the user's `is_active` to false doesn't deactivate the
   technician row ā€” admins can soft-delete the technician row
   independently to keep historical assignments queryable.
6. **`TechnicianCapacitySlot` is per-day, not per-shift.** Multi-shift
   capacity is v1.3 (per `docs/blueprint.md` Ā§21 phase 5). A single
   `(technician_id, day)` row carries the day's total minutes.
7. **APScheduler runs in-process.** No RQ, no Redis ā€” single VPS,
   single-tenant target per `architecture-plan.md` Ā§2.4. Jobs run on
   the production worker (gunicorn) once per app start; tests don't
   start the scheduler unless they explicitly opt in.
8. **Scheduler is opt-in.** `SchedulerConfig.SCHEDULER_ENABLED = True`
   in `ProdConfig`, `False` in `TestConfig` + `DevConfig`. CLI
   commands (`flask run-maintenance-tick`) cover dev triggering.
9. **Maintenance tasks generate eagerly inside a 14-day horizon** at
   each scheduler tick. Longer horizons get noisy lists; shorter
   horizons race the dispatcher.
10. **Translation strategy mirrors 0.6.** Two new modules:
    `maintenance/_translations.py` for task status labels
    (`pending` / `done` / `escalated`) and `planning/_translations.py`
    for assignment-target labels ā€” three short labels each, total ~6
    strings, plus the standard CRUD labels via `lazy_gettext`.

## 3. Options considered

### 3.1 Where the maintenance models live

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A ā€” New `maintenance/` blueprint** (architecture-plan Ā§3.2 mapping) | New blueprint with own `models.py`, `services.py`, `routes.py`, `forms.py`, `templates/maintenance/` | Matches approved architecture plan verbatim; isolates the APScheduler bootstrap; keeps `equipment/` lean | Two new blueprints in one PR is the most code we've shipped in one milestone |
| B ā€” Extend `equipment/` | Maintenance models inside `equipment/models.py` | One fewer blueprint | `equipment/` becomes a fat blueprint; APScheduler bootstrap leaks into a CRUD module; cross-blueprint references (`knowledge ā†’ maintenance`) get harder |
| C ā€” Maintenance + planning in a single `operations/` blueprint | Hybrid | Less surface | The architecture plan calls them out separately; merging them now would have to be un-merged when v0.8 dashboards begin to overlay capacity heatmaps |

### 3.2 What "due" means

| Option | Type | Pros | Cons |
| --- | --- | --- | --- |
| **A ā€” `Date`** (architecture-plan Ā§4) | `Date` column | Matches the cadence ("every 30 days"); no timezone surface area; reports group by day naturally | Loses sub-day scheduling ā€” but v0.7 has no sub-day cadence in scope |
| B ā€” `DateTime(timezone=True)` | timestamp | Future-proofs for hourly cadence | Premature; renders awkwardly ("due 2026-05-13 12:00:00") for monthly plans |
| C ā€” Mixed: `next_due_on` Date + `last_done_at` DateTime | Hybrid | None for v0.7 | Two clock surfaces for one workflow |

### 3.3 APScheduler placement

| Option | Where | Pros | Cons |
| --- | --- | --- | --- |
| **A ā€” `service_crm/shared/scheduler.py`** with a `register_jobs(app)` hook each blueprint calls during `init_app` | Shared module | One place for the BackgroundScheduler instance; jobs declared close to their domain code; opt-in via `SCHEDULER_ENABLED` config | None worth listing |
| B ā€” `service_crm/maintenance/scheduler.py` | Per-blueprint | Locality | Idempotency sweep job (cross-cutting) has no good home |
| C ā€” APScheduler inside each blueprint's `__init__.py` | Distributed | None | Race condition between two blueprints both calling `BackgroundScheduler().start()` |

### 3.4 Task generation horizon

| Option | Strategy | Pros | Cons |
| --- | --- | --- | --- |
| **A ā€” Generate one open task per plan, replenish when previous completes** | Open-set | Keeps the "due tasks" list honest; one row per "due-ness"; matches assumption Ā§2.3 | Doesn't surface "next month's due dates" ā€” but that's reporting, lands in v0.8 |
| B ā€” Generate every task inside a horizon (e.g. 90 days) | Fixed window | Reporting can show the full pipeline | Lots of write churn; tasks get stale if the plan changes; dashboard counts get noisy |
| C ā€” Generate the next two open tasks ("this one + on deck") | Hybrid | A bit of pipeline preview without the noise | Open-set rule Ā§2.3 has to fork; an "on-deck" task that completes out of order is awkward |

### 3.5 Escalation flow

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A ā€” One-click POST** that creates a `ServiceTicket` from the task and stores `task.ticket_id` | Single button | Phone-friendly; idempotency-token guarded; matches roadmap "one-click 'open a ticket from this overdue plan'" wording | Slightly less flexibility ā€” but the user can edit the resulting ticket immediately |
| B ā€” Redirect into `/tickets/new` with pre-filled query params | Two-step | More user control | Two clicks for the common case; if user bails the task stays un-escalated |
| C ā€” Inline modal that builds a ticket draft | Modal | Looks slick | Modal forms break the form_shell contract; over-engineered |

### 3.6 Technician roster vs. User

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A ā€” Separate `technician` row keyed by `user_id`** (architecture-plan Ā§4) | Two tables | Capacity / timezone / planning attrs don't bloat `user_account`; admins can have a row without being technicians | Two lookups for "tech name" ā€” mitigated by `Technician.user` eager-loading in services |
| B ā€” Flag `User.is_technician` + capacity columns on `user_account` | One table | Fewer joins | `user_account` becomes a planning catch-all; capacity columns are NULL for non-technician users |
| C ā€” `user_account` ā†’ no row required, just use `User` everywhere | Zero new tables | Trivial | Loses the capacity table entirely; can't deactivate a technician without disabling their user |

## 6. Open questions

Resolved during implementation; recording the defaults here.

1. **Task `status` enum.** `pending` (just generated), `done` (an
   execution row landed), `escalated` (a ticket was opened). Stored
   as a `String(20)` lookup, not a SAEnum, so adding `cancelled`
   later doesn't require a migration.
2. **Plan with no `last_done_on`.** First `next_due_on` is
   `MaintenancePlan.created_at::date + cadence_days`.
3. **Cascade behaviour.** Plan ā†’ tasks: `CASCADE`. Task ā†’ execution:
   `CASCADE`. Plan ā†’ equipment: `RESTRICT` (deleting equipment with
   live plans is rejected at the service layer).
4. **CSV import for plans / templates.** Out of scope for v0.7; the
   existing `equipment` import pattern can grow it in a patch release.
5. **Permissions.** All routes `@login_required`. Role-based
   restrictions (manager-only template edits) wait for v0.9 hardening.
6. **Audit on `MaintenanceTask`.** Yes ā€” `Auditable`. The state machine
   is two-step (`pending ā†’ done` or `pending ā†’ escalated`) and audit
   coverage is the cheapest insurance.
