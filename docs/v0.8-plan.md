## 1. Audit summary

### 1.1 Mapping to the domain

ROADMAP §0.8.0 closes out the v1 *write* surface (clients → equipment
→ tickets → interventions → maintenance → planning all already exist)
and turns it into a *read* surface for managers and technicians. Per
[`docs/service-domain.md`](./service-domain.md) §"Dashboard V1" and
[`docs/blueprint.md`](./blueprint.md) §13–§14:

`dashboard/` (new blueprint, mounted at `/`):

- Manager view (`/`, `/dashboard/admin`) — KPI tiles + secondary
  panels modelled on `oee-calculator2.0/templates/admin/dashboard.html`.
- Technician view (`/dashboard/me`) — no left sidebar; today's queue,
  one-tap "start intervention", overdue and waiting-parts shortcuts.
  Modelled on `oee-calculator2.0/templates/operator/dashboard.html`.

`reports/` (new blueprint, mounted at `/reports`):

- `tickets_by_status` — tickets grouped by status × period bucket.
- `interventions_by_machine` — count + total minutes per equipment.
- `parts_used` — quantity per part code, scoped to a date window.
- `maintenance_due_vs_completed` — `MaintenanceTask` due count vs.
  `MaintenanceExecution` completed count per period bucket.
- `technician_workload` — interventions, minutes, and completions
  per technician.
- `repeat_issues` — equipment with > 1 ticket in window
  (proxy for the v1.3 "recurring issues" report).

Every report ships an HTML table view + a CSV export at
`/reports/<name>.csv`. CSV stays the v1 export format; spreadsheet
formulas open it natively in RO + EN locales (CSV cells are unquoted
ASCII, headers are translated at render time but stay column-stable).

### 1.2 What's already decided

From [`architecture-plan.md`](./architecture-plan.md) (approved
2026-05-10):

- `dashboard/` is its own blueprint owning **both** the manager and
  technician variants (§6 approval record). Single blueprint, two
  templates — keeps the URL prefix tidy and lets cross-cutting
  helpers (date-window parsing, KPI assembly) live next to each other.
- No new models. Dashboards and reports are pure read views over the
  existing tables. (No migration in this milestone.)
- Layering rule: routes thin, services own the ORM. Same as
  v0.5 → v0.7. Every aggregate query goes through a function in
  `service_crm/<blueprint>/services.py` so the routes stay testable
  against fakes and the dashboard P95 budget has a single seam to
  optimise behind.
- Light mode default, no emoji, no left sidebar on the technician
  view, KPI macro (`macros/kpi_card.html`) shipped in 0.2.0,
  `data_table` + `filter_bar` macros from 0.2.0, locale-aware date
  rendering from 0.1.0.

### 1.3 What v0.8 has to actually decide

1. **One blueprint or two?** Dashboards (cards + panels) and reports
   (tables + CSV) live in different files. They could be one blueprint
   (`dashboard/`) with two URL groups, or two siblings
   (`dashboard/` + `reports/`). See §3.1.
2. **Where the home page lives.** `/` currently has no owner (it
   404s; the sidebar's "Dashboard" link is `href="#"`). Either
   `dashboard/` claims `/` or we keep `/` empty and mount the
   manager view at `/dashboard`. See §3.2.
3. **Manager vs. technician routing.** Auto-route based on
   `current_user.role.name`, or always serve the manager view and
   provide a "switch to my queue" link that goes to
   `/dashboard/me`. See §3.3.
4. **KPI computation: per-request or cached?** Compute on every
   request (simple, correct, possibly slow on the reference dataset)
   vs. cache in-process with a TTL, vs. precompute via an
   APScheduler tick. See §3.4.
5. **Date-window parser.** Each report needs `from` / `to`. Build
   a tiny `shared/date_window.py` helper, or duplicate the
   ISO-date parsing already in `planning/routes.py`. See §3.5.
6. **CSV export shape.** One `export_csv` helper that takes
   `(columns, rows)` vs. per-report custom writers. See §3.6.
7. **"Today's queue" definition for the technician view.** Tickets
   assigned to me that are not closed/cancelled, plus pending
   maintenance tasks assigned to my Technician row — what counts
   as "today"? See §3.7.

## 2. Assumptions

1. **No new tables, no migration.** Every aggregate is a read query
   over the v0.7 schema. If a report needs a column that doesn't
   exist (e.g. SLA-breached count), it's out of scope for 0.8 and
   gets a follow-up issue, not a schema patch.
2. **Aggregates compute on every request.** Caching is deferred to
   0.9 hardening (§1.3 perf budget validation) — if the reference
   dataset's 300 ms P95 is missed there, we add a TTL cache then.
   Premature caching this milestone hides correctness bugs.
3. **Default date window is the current month-to-date.** Manager
   dashboard tiles read "this month" by default; the date-range
   picker can widen or narrow. Reports default to "this month"
   per [`docs/v1-implementation-goals.md`](./v1-implementation-goals.md)
   §0.8 ease-of-use bar.
4. **CSV export = RFC 4180 with `\r\n` line endings, UTF-8 with no
   BOM, comma-separated, headers translated.** No BOM keeps the
   file diffable; Excel handles UTF-8 CSV natively since 2017.
5. **The technician dashboard reads the technician's `Technician`
   row, not their `User` row.** A user without a `Technician` row
   sees an empty queue with a hint to ask an admin to provision
   one. (Same behaviour as the planning capacity grid in v0.7.)
6. **No role-based redirect on `/`.** Both managers and technicians
   land on the manager view; technicians click the "My queue" link
   in the topbar or sidebar to reach `/dashboard/me`. Role-based
   landing is a v0.9 hardening item (§1.4 RBAC matrix).
7. **All routes are `@login_required`.** Role-based restrictions
   (e.g. manager-only CSV export) wait for v0.9.
8. **"Open ticket" = any ticket whose status is not in
   `{closed, cancelled}`.** Matches the existing `list_tickets(open_only=True)`
   filter.
9. **"Overdue ticket" = open ticket with `due_at < now`.** SLA
   breaches are out of scope until the SLA columns get populated by
   a real workflow (v0.9 hardening per §1.4).
10. **"Due maintenance this week" = `MaintenancePlan.is_active AND
    next_due_on BETWEEN start_of_week AND end_of_week`.** Week
    starts Monday (ISO 8601), matching the technician roster's
    Romanian default locale.
11. **"Technician utilization" = (scheduled minutes / declared
    minutes) over the current week**, where scheduled minutes use
    the same `DEFAULT_ASSIGNMENT_MINUTES = 60` constant
    `planning/services.py` uses today. Renders as a percentage tile.
12. **Translation strategy mirrors 0.7.** Per-blueprint
    `_translations.py` modules expose the report code → label map
    via `lazy_gettext`. CSV column labels go through the same map
    so server-side translation picks them up at request time.

## 3. Options considered

### 3.1 One blueprint or two

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Two siblings (`dashboard/` + `reports/`)** | Two blueprints, two URL prefixes (`/`, `/reports/`) | Reports own `reports/services.py` (pure aggregation) and `reports/routes.py` (HTML + CSV) — the dashboard composes them via cross-blueprint service calls; the two surfaces evolve independently; matches the v0.5/0.7 "one blueprint per slice" precedent | One extra blueprint registration; sidebar gets a new entry |
| B — Single `dashboard/` blueprint with `routes_reports.py` | One blueprint, two route modules | Single URL prefix | The two surfaces have different cache horizons (KPI tiles change minute-to-minute, reports change day-to-day), packing them into one blueprint loses that separation; CSV download routes get a confusing `dashboard.reports_csv` endpoint name |
| C — Reports inside each existing blueprint (tickets, maintenance, …) | Distributed | Locality | Same "where does the dashboard pull from" question that's already in the architecture plan — the central blueprint loses; cross-blueprint joins (per-tech workload spans tickets + interventions + maintenance) have no clean home |

### 3.2 Where the home page lives

| Option | Mounting | Pros | Cons |
| --- | --- | --- | --- |
| **A — `dashboard/` owns `/`** | `bp = Blueprint("dashboard", __name__)` mounted at `/` with rules `/` and `/dashboard/me` | Matches user expectations ("the home page is the dashboard"); sidebar's `href="#"` placeholder gets a real URL; preserves `/dashboard/admin` for explicit links | `dashboard/` doesn't have a URL prefix so its routes look different from the other blueprints |
| B — `dashboard/` mounts at `/dashboard`, `/` redirects | Two surfaces | Each blueprint has a URL prefix | Two endpoints to keep in sync; the unauthenticated `/` (which the auth blueprint redirects to login from) gains an indirection |
| C — `dashboard/` mounts at `/dashboard`, `/` 404s | Status quo | No work | Sidebar still has `href="#"`; bad UX |

### 3.3 Manager vs. technician routing

| Option | Default | Pros | Cons |
| --- | --- | --- | --- |
| **A — Always serve manager view at `/`; `/dashboard/me` for the technician view; both linked from topbar/sidebar** | Manager-first | Predictable URL space; bookmarkable; both views available to every authenticated user (managers may still want to check the queue) | A technician's first visit lands on a page denser than they need — mitigated by a "My queue" pill at the top of the manager view |
| B — Role-based redirect on `/` | Conditional | Technicians get their queue immediately | Role-based behaviour belongs to v0.9 hardening (§1.4 RBAC matrix); doing it now adds a redirect that breaks deep links; testing the redirect requires every test to know its role |
| C — Single dashboard that toggles via a query arg | One URL | Fewer endpoints | Conditional rendering of completely different layouts is the classic "fat handler" smell; CSV exports get a `?view=` argument that's easy to forget |

### 3.4 KPI computation strategy

| Option | Strategy | Pros | Cons |
| --- | --- | --- | --- |
| **A — Per-request compute, no cache** | Each route call hits the DB | Always correct; one obvious seam to optimise behind (the service function); simple to test (no cache invalidation) | If the reference dataset busts the 300 ms budget we add caching in 0.9 — but the budget is a 0.9 goal, not a 0.8 one (per [`v1-implementation-goals.md`](./v1-implementation-goals.md) §1.3) |
| B — In-process TTL cache | `functools.lru_cache` with a 60s TTL wrapper | Free P95 win | Cache invalidation on writes is awkward (we'd need every service to invalidate by name); per-process means multi-worker deployments serve stale data; complicates tests |
| C — Precompute via APScheduler tick | Materialised tiles in a `dashboard_snapshot` table | Sub-10ms tile reads | Adds a model + migration (out of scope per Assumption §2.1); precompute lag breaks the "drillable from a tile" invariant in the ease-of-use bar |

### 3.5 Date-window parser

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Tiny `shared/date_window.py`** | `parse_window(args, *, default_days=30) -> tuple[date, date]` + presets (`this_week`, `this_month`) | One implementation, one place to test; reusable from dashboard + reports + any 1.0 hardening report we add later | Adds one module to `shared/` — but `shared/clock.py` already sets that precedent |
| B — Duplicate the `_parse_date` helper from `planning/routes.py` | Copy-paste | No new module | Three copies in three blueprints = three bug surfaces; the planning helper doesn't support presets |
| C — Use WTForms `DateField` even for read-only filters | Form-based | Free validation | Read-only GET URLs shouldn't need a CSRF-protected form; bookmarking gets ugly |

### 3.6 CSV export shape

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — One `reports/csv.py::write_csv(headers, rows) -> Response`** | Shared helper | Same `Content-Type`, same encoding, same line-endings, same filename pattern (`<report>-<from>-<to>.csv`); each report wires `(translated_headers, list_of_tuples)` and that's it | Slightly less per-report flexibility — but reports stay structurally similar by design |
| B — Per-report custom writers | Locality | Each report can format its own cells | Six near-duplicates; encoding bugs surface six times |
| C — Stream from a generator | Memory-efficient | Handles 1M-row exports | Reference dataset is < 100k rows per report; the simpler in-memory writer fits inside Flask's default response and avoids the streaming-response edge cases that always bite during 0.9 hardening |

### 3.7 "Today's queue" definition for the technician view

| Option | Definition | Pros | Cons |
| --- | --- | --- | --- |
| **A — `(tickets assigned to me, not terminal) ∪ (maintenance tasks assigned to my Technician row, pending)`** | Union | Captures both event-driven (tickets) and planned (maintenance) work; matches the v1 workflow described in [`blueprint.md`](./blueprint.md) §9 | Two queries; ranking has to interleave by date |
| B — Tickets only | Single source | Trivial | Hides the "you have a maintenance task today" case — the entire reason the v0.7 milestone shipped the link |
| C — Anything where I'm the named technician (assignments, interventions, tasks) | Three sources | Most complete | Open interventions are an "in flight" state, not a queue entry; surfacing them duplicates the parent ticket card |

## 4. Recommendation

**Pick A on every option** (§3.1A through §3.7A). All A choices match
the approved architecture plan, the 0.5 / 0.6 / 0.7 patterns, and the
ease-of-use bar in §0.8 of `v1-implementation-goals.md`.

## 6. Open questions

Resolved during implementation; recording the defaults here.

1. **Period bucketing for `tickets_by_status` and
   `maintenance_due_vs_completed`.** Day for windows < 31 days,
   week for windows 31–180 days, month for everything longer.
   Buckets computed in Python after the SQL pull — the reference
   dataset is small enough that SQL-side `date_trunc` isn't worth
   the SQLite / Postgres branching.
2. **"High-risk machines" definition.** Equipment with ≥ 3 tickets
   opened in the dashboard's current window. Tunable later; a
   constant in `dashboard/services.py` for now.
3. **"Recent interventions" cap.** 10 rows, ordered by
   `started_at DESC`. Matches the maintenance task list pattern.
4. **CSV filename pattern.** `<report-code>-<from>-<to>.csv`,
   where `from` / `to` are ISO dates with no separators
   (e.g. `tickets-by-status-20260501-20260513.csv`). Stable across
   locales.
5. **CSV header translation.** Headers go through `gettext` at
   render time, but the *codes* remain English (the stable
   identifier per [`blueprint.md`](./blueprint.md) §14 "Internal raw
   codes remain stable"). Both ship: column 1 is the code, column
   2 is the translated label, columns 3+ are the data. Spreadsheet
   pivots use column 1; humans read column 2.
6. **Audit log on dashboard / reports.** No — these are pure reads.
   No `Auditable` mixin, no audit events. Matches the lookup-table
   precedent (`TicketPriority`, `TicketType`) from 0.5.
