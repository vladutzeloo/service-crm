## 1. Audit summary

### 1.1 Mapping to the domain

ROADMAP §0.5.0 is the **first** of two ticket slices. v0.5 lands the
ticket *spine*; v0.6 hangs interventions, parts, and knowledge off it.
Per [`service-domain.md`](./service-domain.md) "Ticket Domain" +
[`architecture-plan.md`](./architecture-plan.md) §4, this slice owns
five new tables in a new `tickets/` blueprint:

- `service_ticket`
- `ticket_status_history`
- `ticket_comment`
- `ticket_attachment`
- the lookups `ticket_type` and `ticket_priority`

The state machine (`tickets/state.py`) is **pure Python** and is the
only new sub-module beyond the standard `models / services / routes /
forms` quartet.

### 1.2 What's already decided

From [`architecture-plan.md`](./architecture-plan.md) (approved
2026-05-10):

- Model shapes (§4) and constraint list (§4 closing bullets).
- Stable English `code` columns on `TicketType` / `TicketPriority` with
  translated UI labels ([`v1-implementation-goals.md`](./v1-implementation-goals.md)
  §3.2).
- `TicketStatusHistory` is the audit-of-record for state changes —
  **no `Auditable` mixin** on it (it *is* the audit).
- `from_state IS NULL` marks creation; the existing pattern note in
  arch-plan §4 must be obeyed by list/filter queries.
- ULID PKs across the board (matches `clients` + `equipment`).

### 1.3 What v0.5 has to actually decide

The model schema is sketched, but five execution-level questions are
still open:

1. **Where the state-transition guard lives** (pure-Python FSM vs.
   SQLAlchemy validators vs. service-only). See §3.1.
2. **How the history row is written** (`before_flush` hook on
   `ServiceTicket.status` vs. explicit `services.transition_ticket`
   write vs. DB trigger). See §3.2.
3. **`ticket.number` generation under SQLite** — flagged in
   [`architecture-plan.md`](./architecture-plan.md) §4 "race-condition
   note". v0.5 has to actually pick a resolution. See §3.3.
4. **Attachment storage path + MIME validation** (filesystem under
   `instance/uploads/` with magic-byte sniffing, per
   [`v1-implementation-goals.md`](./v1-implementation-goals.md) §1.4).
   See §3.4.
5. **Idempotency-token persistence model** (dedicated `idempotency_key`
   table vs. piggybacked on the form session). See §3.5.

## 2. Assumptions

Numbered so they can be shot down individually. Each is falsifiable.

1. **Interventions are NOT in v0.5.** `ServiceIntervention`,
   `InterventionAction`, `InterventionFinding`, `PartMaster`,
   `ServicePartUsage` ship in v0.6. The
   `TicketAttachment.intervention_id` FK column in arch-plan §4 is
   deferred to v0.6 — adding it now would require either a stub
   `service_intervention` table or a nullable FK to a non-existent
   table. *Falsify by:* showing a v0.5 requirement that needs
   interventions.
2. **The status enum is closed.** The nine values in
   [`service-domain.md`](./service-domain.md) "Ticket Lifecycle" are
   the v1 set; we don't introduce per-tenant custom statuses.
   *Falsify by:* product saying "we need a `parked` status for site 3".
3. **`TicketType` and `TicketPriority` are admin-managed lookups,
   not free-text.** Seeded with stable codes (`incident`, `preventive`,
   `commissioning`, `warranty`, `installation`, `audit` for type;
   `low / normal / high / urgent` for priority). UI translates labels
   via `_()` against the `code`. *Falsify by:* a requirement to add
   types from inside the ticket form.
4. **Manager qualifies / schedules; technician executes.** The form
   for `new → qualified` (type + priority + SLA + assignment) is the
   *manager* form; the technician's `in_progress → waiting_parts /
   monitoring / completed` form is simpler. RBAC is enforced by route
   decorators (existing pattern, not new wiring).
5. **Attachments are files on local disk under
   `instance/uploads/<ticket_hex>/<ulid>.<ext>`.** No S3, no object
   store in v1 — single-VPS posture from
   [`../ARCHITECTURE.md`](../ARCHITECTURE.md) §3. MIME validated by
   extension *and* magic bytes (Pillow for images, `mimetypes` for the
   rest). *Falsify by:* capacity / multi-host requirement.
6. **Comments are plain text.** No Markdown rendering, no mentions, no
   edit history. Free-text only; size capped at 8 KB to keep the audit
   `before/after` JSON sensible. *Falsify by:* product needing rich text.
7. **One actor per transition.** `TicketStatusHistory.actor_id` is
   non-nullable; `current_user` must be authenticated. System
   transitions (e.g. APScheduler closing tickets) are v0.7+ and will
   need a `system` user row.
8. **SLA fields are scalar columns on `ServiceTicket`**
   (`sla_due_at: DateTime`), not a separate `ServiceLevelAgreement`
   model. `ServiceContract` lives in `clients/`; v0.5 doesn't touch it.
9. **Search lights up the same FTS pattern as clients/equipment**
   (PG `tsvector` + GIN on `title || description`; SQLite `LIKE
   lower()` fallback). No new tokenizer config.
10. **Translation strategy for status / type / priority labels = code
    lookup in templates**, e.g.
    `{{ _(ticket.status.value | capitalize) }}` with translatable
    strings registered in a `tickets/_translations.py` module so
    `pybabel extract` catches them. **No translation table.**
11. **Idempotency tokens are already wired into `form_shell`**
    (per `CHANGELOG.md` 0.2.0 — "the directory carries the
    idempotency-token hidden input"). v0.5 adds the *server-side*
    dedup, not the client field.

## 3. Options considered

### 3.1 State-transition guard placement

| Option | Where the legality check runs | Pros | Cons |
| --- | --- | --- | --- |
| **A — Pure-Python FSM in `tickets/state.py`** (the roadmap's wording) | `state.transition(from, to, role) → to_state \| raise` called from `services.transition_ticket` | Hypothesis state-machine test maps cleanly onto pure-Python transitions; mockable; no DB hit; reusable from CLI / future API; matches `≥ 95 %` coverage gate trivially | Two layers must agree (service must call it; nothing physical stops a future dev from `ticket.status = "closed"` directly) |
| B — SQLAlchemy `@validates("status")` on the model | Reject illegal transitions at attribute set | Hard to bypass | Couples FSM to ORM; can't be tested without a session; awkward to inject the actor/role; gating role-based moves at this layer leaks RBAC into models |
| C — DB CHECK constraint + trigger | Enforced at the storage layer | Truly impossible to bypass | Postgres-only triggers; SQLite parity becomes a chore; FSM stateful tests must run against a real DB; legibility tanks |

### 3.2 How `TicketStatusHistory` rows get written

| Option | Mechanism | Pros | Cons |
| --- | --- | --- | --- |
| **A — `before_flush` SA hook on the `db.session`** (roadmap wording: "`before_flush` hook writes a row on every status change") | Inspect `session.dirty` for `ServiceTicket` instances whose `status` is in the change set; build a `TicketStatusHistory` from `inspect(t).attrs.status.history` | Bulletproof — every status change, regardless of code path, produces history; mirrors the existing `Auditable` pattern in `shared/audit.py` | Needs `ACTOR_CTX` set; same-session subtleties (the new row must be added to the flush in progress) require careful ordering |
| B — Service-layer only: `services.transition_ticket(...)` explicitly inserts a history row | Easy to reason about | Easy to forget; a fixture or admin script that writes `ticket.status = …` bypasses history. ROADMAP acceptance criterion *"integration tests assert no history-less status transitions"* would have to test every call site instead of the invariant. |
| C — DB trigger | — | Same as 3.1 C — Postgres-only complexity. |

### 3.3 `ServiceTicket.number` generation

[`architecture-plan.md`](./architecture-plan.md) §4 **explicitly leaves
this open** for the 0.5 implementer. Choices:

| Option | Mechanism | Pros | Cons |
| --- | --- | --- | --- |
| **A — `Sequence("ticket_number_seq")` on PG; service-layer SQLite fallback with `SELECT … FOR UPDATE`-style serialisation** | Native sequence on PG; on SQLite serialise `services.create_ticket` via a row-level lock on a sentinel row | Human-friendly monotonic number; no enumeration concerns (ULID is the PK) | SQLite serialisation needs to be tested explicitly; small extra complexity in `services.create_ticket` |
| B — Drop `number` entirely, show `ticket.id[:8]` as the human label | Zero extra schema | "TIX-4F8A2C1D" isn't operator-friendly; managers will hate it |
| C — Add a separate autoincrement surrogate PK and keep ULID as a secondary id | DB-managed integer | Loses ULID as the single edge identifier; breaks symmetry with clients/equipment |

### 3.4 Attachment storage

| Option | Where bytes live | Pros | Cons |
| --- | --- | --- | --- |
| **A — Filesystem under `instance/uploads/tickets/<ticket_hex>/<attachment_ulid>.<ext>`**, metadata in `ticket_attachment` | One container, one volume mount; cheap; mirrors the `v1-implementation-goals.md` §2.4 "stored outside the static path" rule | Backup script needs to include the upload tree (already called out in `v1-implementation-goals.md` §1.6) |
| B — BYTEA in `ticket_attachment.body` | One backup unit | Bloats DB; row size limits get hit; row-level cache pollution; restore is slow |
| C — S3 / object store | Scales | Breaks the "one VPS" posture; out of scope for v1 |

### 3.5 Idempotency dedup persistence

| Option | Storage | Pros | Cons |
| --- | --- | --- | --- |
| **A — Dedicated `idempotency_key` table: `(user_id, token) UNIQUE, route, result_url, expires_at`, swept after 24 h** | DB | Reproducible across worker restarts; testable; works on both DBs; can return the prior result URL on retry (matches `v1-implementation-goals.md` §2.5) | One extra small table |
| B — Redis | Cache | Reintroduces Redis as a hard dep; `ARCHITECTURE.md` §3 says "later: RQ + Redis" |
| C — In-process LRU | Memory | Fails the moment we run more than one worker |

### 3.6 Slicing (per-PR ordering)

| Option | Shape | Pros | Cons |
| --- | --- | --- | --- |
| **A — Three PRs: (1) models + state machine + history + lookups + seeding + tests; (2) routes + forms + list + detail + filters + idempotency; (3) comments + attachments + photo upload pipeline + i18n closeout** | Reviewable, mergeable, each is green on `main` | One milestone but three release-notes lines |
| B — One mega PR | Atomic | ~4 k LOC review; high bus-factor risk |
| C — Five micro PRs (split lookups, history, comments, attachments, list/detail) | Tiny diffs | Synchronization overhead between PRs; thrashes the migration file |

### 3.7 Where the ease-of-use bar lives (project-level)

Resolved in this plan; restated for the record.

| Option | Where | Pros | Cons |
| --- | --- | --- | --- |
| **A — Extend `v1-implementation-goals.md` with a new §3.5** | Same file as a11y / i18n | One source of truth; release exit checklist (§5) automatically picks it up; matches the existing pattern reviewers already know | Touches the big goals doc |
| B — New `docs/ease-of-use.md` standalone | New file | Easier to evolve in isolation | Risks orphaning; reviewers won't think to read it |
| C — Bake into each `blueprint.md` slice individually | Distributed | Closest to the code | Drifts; no global bar; bad for grading releases |

### 3.8 How the bar is enforced per PR

| Option | Mechanism | Pros | Cons |
| --- | --- | --- | --- |
| **A — Hybrid: a11y / Lighthouse / touch-targets in CI; the rest via a `/ease-pass` skill checklist + reviewer signoff** | Mixed | Pragmatic; matches how a11y and i18n already work | Two failure modes (skill vs. CI) — but the team is already used to that |
| B — Mechanise everything via Playwright assertions in CI | Auto | Catches regressions for free | Some checks (smart defaults; "do I leave the page?") are awkward to express as assertions; high test-maintenance cost |
| C — Manual reviewer checklist only | Manual | Cheapest | No regression protection; relies on memory |

## 4. Recommendation

**Pick A on every option** (§3.1A through §3.8A). Rationale: every "A"
matches an explicit line in the roadmap, the architecture plan, or the
v1 implementation goals — and every "A" is consistent with the patterns
already shipped by `clients` and `equipment` (`Auditable` mixin,
service-layer guards, instance-folder storage). Deviating would
re-open closed decisions.

### 4.1 Per-slice ordering (the actual plan)

#### Slice 0.5.a — Models, state machine, history, lookups

**Goal:** ship the ticket spine without any UI. Routes return JSON or
template stubs; integration tests drive everything. **Size:** ~1.3 k
LOC; ~50 tests.

1. `service_crm/tickets/__init__.py` (blueprint stub, mounted but
   routes return 404 for now).
2. `service_crm/tickets/state.py` — pure-Python FSM. Functions:
   - `legal_transitions(from_state, role) → set[TicketStatus]`
   - `transition(from_state, to_state, role) → TicketStatus | raise IllegalTransition`

   Hypothesis state-machine test sits beside it
   (`tests/tickets/test_state.py`).
3. `service_crm/tickets/models.py`:
   - `TicketStatus` (Python enum, stable English values)
   - `TicketType` (lookup, with `is_default: bool` for ease-of-use
     smart defaults — see §4.2)
   - `TicketPriority` (lookup, with `is_default: bool`)
   - `ServiceTicket` (with `number` via `Sequence("ticket_number_seq")`
     on PG, service-layer fallback on SQLite)
   - `TicketStatusHistory` (no `Auditable` mixin)
4. `migrations/versions/<ts>_<rev>_tickets_spine.py` — creates all
   four tables, the sequence on PG, the GIN expression-index on
   `title || description` for PG search, the indices on `status`,
   `client_id`, `equipment_id`, `priority_id`, `due_date`, plus seeds
   for `ticket_type` and `ticket_priority` (defaults from
   [`blueprint.md`](./blueprint.md) §10 / §12). Down-migration tested
   by `pytest-alembic` (already in CI from 0.9 onwards; here we just
   write a clean `downgrade()`).
5. `service_crm/tickets/services.py`:
   - `create_ticket(...)` — inserts ticket + writes the
     `from_state=NULL, to_state="new"` history row in the same
     transaction.
   - `transition_ticket(ticket, to_state, actor, reason)` — calls
     `state.transition` then sets `ticket.status` (the `before_flush`
     hook produces the history row).
   - Lookup CRUD for `TicketType` / `TicketPriority` (same shape as
     `EquipmentControllerType` in `equipment/services.py`).
   - `_validate_equipment_belongs_to_client` — mirrors
     `_validate_location_belongs_to_client` from `equipment`.
6. **`before_flush` listener** lives in `service_crm/shared/audit.py`
   (alongside `Auditable`) — it's the same kind of hook, and
   centralising keeps the audit listener registration in
   `_register_audit_listeners` from `__init__.py` working without
   changes. The hook reads `ACTOR_CTX` for the actor id; if missing,
   raises (no anonymous transitions in v0.5).
7. Tests: models (table shape, unique/CHECK constraints, FK cascades,
   `is_active` semantics), state machine (≥ 95 % line + branch,
   Hypothesis state-machine), services (legal/illegal transition
   cases, history row written exactly once, cross-blueprint guard,
   ticket number race on SQLite under threaded writes).

3. `service_crm/tickets/forms.py` — `TicketCreateForm`,

   `TicketFilterForm`. All extend a `_IdempotentForm` base that emits

4. `service_crm/tickets/routes.py`:
   - `/tickets/`
   - `/tickets/new`
   - `/tickets/<hex>`
   - `/tickets/<hex>/edit`
   - `/tickets/<hex>/qualify`
   - `/tickets/<hex>/schedule`
   - `/tickets/<hex>/transition`
   - `/tickets/<hex>/cancel`
   - `/tickets/bulk/<action>` (priority, cancel, assign-stub)
   - lookup routes (`/tickets/types[/...]`, `/tickets/priorities[/...]`)
   - `/clients/<hex>/new-ticket`, `/equipment/<hex>/new-ticket`

5. Templates (mirroring `equipment/`):
   - `tickets/list.html` — filter bar (status, type, priority, due,

6. Sidebar `templates/base.html:32` link points at `/tickets/` (drop
   the `#`). Topbar gains "My queue" link for users with the
   technician role.
7. Tests: routes (full e2e for each route including idempotent-retry,
   403 on cross-blueprint shenanigans, filter combinations), services
   (filter SQL on both DBs), Playwright smoke (the 0.5.0 line in

#### Slice 0.5.c — Comments, attachments, i18n closeout, draft auto-save

**Goal:** the ticket detail page becomes a real timeline; the
photo-upload pipeline first shipped in 0.4 is tied to a ticket.
**Size:** ~1.1 k LOC; ~40 tests.

1. New migration: `ticket_comment`, `ticket_attachment` (no
   `intervention_id` FK yet — that column is added in 0.6 when
   `service_intervention` exists).
2. `service_crm/shared/uploads.py` — first real consumer of the
   pipeline. Validates MIME (extension + magic bytes via Pillow for
   images and `mimetypes` for the rest), compresses images (long edge
   ≤ 2048 px, WebP q85 per `v1-implementation-goals.md` §2.4), writes
   under `instance/uploads/tickets/<ticket_hex>/<attachment_ulid>.<ext>`,
   returns the storage key. Same pipeline is reused by 0.6
   interventions.
3. `service_crm/tickets/services.py` — `add_comment(...)`,
   `add_attachment(...)`, `delete_attachment(...)` (soft-delete via
   `is_active` to preserve the audit chain). Attachments use a private
   `/tickets/<hex>/attachments/<aid>` route that streams from disk
   after an auth check — **never** linked from `static/`.
4. Detail template: comments rendered chronologically via `data_table`
   in stacked-card form; attachments rendered as a thumbnail grid with
   same-aspect-ratio thumbs (matches `v1-implementation-goals.md`
   §2.4). Comment compose box is sticky at the bottom on mobile;
   below the timeline on desktop.
5. **Reason field is required for `cancel`** and for `delete
   attachment` (forgiving-workflow check from §3.5 of
   `v1-implementation-goals.md`).
6. `service_crm/static/js/app.js` — ~30 LOC `sessionStorage` draft
   auto-save module, keyed by the idempotency token. No server state;
   pure progressive enhancement.
7. Full i18n pass: every new string in `_()` / `{% trans %}`;
   `pybabel extract` + `pybabel update`; both `messages.po` files
   updated; `pybabel compile` clean. Status / type / priority codes
   translated via a single `tickets/_translations.py` keyed by `code`,
   so `{{ _(ticket_type_label(ticket.type.code)) }}` is the single
   call site.
8. Tests: comments (route e2e, body cap, audit row written),
   attachments (upload happy path, oversize image rejected, magic-byte
   mismatch rejected, private fetch requires auth, soft-delete
   behavior), i18n (`pybabel extract` no-diff check at minimum),
   draft auto-save round-trip (Playwright on Pixel-5).

| Phone ergonomics | Technician | At 360 px portrait: bottom-fixed full-width action bar (primary action within thumb reach); status pills wrap, never scroll horizontally; tap targets ≥ 44 pt (already enforced) | 0.5.b |

The v0.5 cost of the bar: +1 column × 2 lookups, +1 model
(`UserTicketFilter`), +1 route family (`/tickets/bulk/<action>`),
+~30 LOC JS, +~5 e2e tests, +~200 LOC service / route. **~15 % bigger
than the bar-less version**, inside the milestone budget.

## 6. Open questions

These need answers before slice 0.5.b opens (slice 0.5.a doesn't
depend on any of them; we can start there in parallel).

1. **Ticket numbering format.** Raw integer (`1`, `2`, `3`, …) or a
   prefixed display (`TIX-000123`, `T-2026-001`)? Storage is the same;
   this is a `format_ticket_number(n)` helper.
   *Default if unanswered:* zero-padded integer with a `T-` prefix at
   display time.
2. **Initial assignment at creation.** Can a front-desk user create a
   ticket already in `qualified` state (skipping the `new`
   qualification step) if they have all the fields? Or is `new →
   qualified` always a separate manager action?
   *Default:* always separate, mirrors
   [`service-domain.md`](./service-domain.md) "Ticket Lifecycle"
   wording.
3. **SLA semantics.** Is `sla_due_at` a *target* (informational) or a
   *gate* (a ticket past SLA can't be `closed` without an override
   reason)?
   *Default:* informational in v0.5; gate behaviour deferred to v0.7
   when reporting lands.
4. **Cancellation reason.** Is `TicketStatusHistory.reason` required
   for `cancelled` transitions?
   *Default:* required for `cancelled`, optional everywhere else —
   checked in `services.transition_ticket`, not at the DB layer.
5. **Comment authoring on closed tickets.** Allowed (audit trail
   useful) or forbidden (immutable post-close)?
   *Default:* allowed; the audit row records who commented after close.
6. **Attachment size cap.** Single file size limit (e.g., 25 MB) and
   per-ticket aggregate cap (e.g., 200 MB)? Needed for the upload
   pipeline test matrix.
   *Default:* 25 MB single / no aggregate cap in v0.5; revisit in v0.9.
7. **Idempotency window persistence across deploys.** Should the
   `idempotency_key` rows survive a container restart?
   *Default:* yes — they live in Postgres, swept by a `flask` command
   that runs out of `cron` or APScheduler from 0.7.
8. **Lookup-table editing UI.** Mirror equipment's
   `controllers_list.html` / `models_list.html` pattern (CRUD + CSV
   import)? Or admin-only CLI seeding in v0.5 and UI in v0.7?
   *Default:* mirror equipment's pattern in slice 0.5.b — it's already
   in place; not doing it would leave the page stubbed.
9. **`TicketAttachment.intervention_id` shape now or later.** Add the
   nullable column in slice 0.5.c (pointing at a not-yet-existing
   table) or add it in v0.6 alongside the `service_intervention`
   table?
   *Default:* defer to v0.6 — adding a FK to a table that doesn't
   exist is awkward on SQLite and creates a no-op constraint on PG.
10. **One PR per slice vs. one PR for v0.5.** Three is reviewable;
    one is faster for the contributor.
    *Default:* three; ROADMAP says "0.5 stays reviewable" verbatim.
11. **Inline modal vs. full-page for `qualify` and `schedule`.**
    Manager flows tend to be field-heavy.
    *Default:* modal for `schedule` (small) + `cancel` (tiny) +
    `transition` (one click); dedicated page for `qualify` (4–5 fields
    with their own validation).
12. **Bulk-action scope.** v0.5 ships `bulk priority`, `bulk cancel`,
    `bulk assign` (stubbed). Do we also want `bulk close` and
    `bulk re-open`?
    *Default:* no — closing and re-opening are status moves that
    deserve per-ticket reason fields.
13. **Saved-filter cap.** Per-user limit on saved filters before the
    dropdown gets unwieldy?
    *Default:* 8 per user, oldest evicted; trivially adjustable.
14. **"My queue" filter membership.** Includes `scheduled` and
    `in_progress`. Does it also include `waiting_parts` (the
    technician is blocked but still owns it)?
    *Default:* yes — exclusion hides work the technician needs to be
    reminded of.
15. **Draft auto-save scope.** Just the ticket create form, or every
    form in `tickets/`?
    *Default:* every form — the JS is generic.
16. **Reason wording.** Required reason for `cancel` and `close` —
    "free text mandatory" or "choose from a dropdown of common reasons
    + optional text"?
    *Default:* dropdown of 6 common reasons + a "type your own"
    option; the chosen code lands in
    `TicketStatusHistory.reason_code`, the free-text in `reason`.
17. **Which dashboard tiles drill down in v0.5 vs. 0.8.** v0.8 owns
    the dashboard, but the tickets list link can already exist.
    *Default:* the link from any "open tickets" / "overdue tickets"
    tile is wired in 0.8; v0.5 just guarantees the URL fragment
    (`/tickets/?status=overdue`) works.
