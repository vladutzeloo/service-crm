# Data Model

Codifies the "Data Model" workflow from
[`docs/tasks.md`](../../../docs/tasks.md).

## Inputs (read these first)

1. [`AGENTS.md`](../../../AGENTS.md) — architecture rules (SQLAlchemy +
   Alembic, services own the ORM, migrations reviewable + reversible).
2. [`docs/service-domain.md`](../../../docs/service-domain.md) — the
   entities and relationships you are modeling.
3. [`ARCHITECTURE.md`](../../../ARCHITECTURE.md) §4 — layout, layering
   rules, cross-cutting (ULIDs, audit log, money, time).
4. [`python.tests.md`](../../../python.tests.md) — fixture and factory
   patterns you'll need.
5. [`docs/architecture-plan.md`](../../../docs/architecture-plan.md) §4 —
   the agreed model sketches.

## What this skill produces

For the requested module(s):

- `service_crm/<bp>/models.py` — SQLAlchemy 2.x models. Inherit from
  `db.Model` and the `Auditable` mixin (in `service_crm/shared/audit.py`).
- A new revision under `migrations/versions/` (created via
  `flask --app service_crm db migrate -m "..."`, then **edited**).
- `tests/<bp>/test_models.py` and `tests/<bp>/test_services.py` covering
  relationships, unique constraints, cascade behavior, and any
  service-layer guard rules.
- `tests/factories.py` updates — one factory per new model.

## Procedure

1. **Read the agreed sketch** in
   [`docs/architecture-plan.md`](../../../docs/architecture-plan.md) §4. If
   the change is not there, **stop and run** `/architecture-audit` first.
2. **Write the models** in `service_crm/<bp>/models.py`. Conventions:
   - PK column is `id = mapped_column(ULID, primary_key=True,
     default=ulid_new)`.
   - Use `mapped_column` (SQLAlchemy 2.x style), not the legacy
     `Column(...)` form.
   - FKs name the target column explicitly: `ForeignKey("client.id",
     ondelete="CASCADE")`.
   - `created_at`, `updated_at` — `DateTime(timezone=True)`, defaulted via
     `service_crm.shared.clock.now`.
   - JSON columns use `db.JSON` (works on PG and SQLite); document the
     shape in a docstring.
   - Add `__table_args__` for composite indexes / check constraints.
3. **Generate the migration**:
   ```powershell
   flask --app service_crm db migrate -m "add <module> tables"
   ```
4. **Review the autogenerated revision** before committing:
   - Add `op.execute(...)` for any data backfill.
   - Add `server_default=` for `NOT NULL` columns on populated tables.
   - Confirm the downgrade is real (drops what upgrade added).
   - If the dev DB is SQLite, ensure `render_as_batch=True` is set in the
     Alembic env.
5. **Apply and round-trip** locally:
   ```powershell
   flask --app service_crm db upgrade
   flask --app service_crm db downgrade -1
   flask --app service_crm db upgrade
   ```
6. **Tests**, in this order:
   - `test_models.py`: relationships load both sides, cascade deletes
     work, unique constraints reject duplicates.
   - `test_services.py`: any guard rules from `docs/architecture-plan.md`
     §4.1 (e.g., "`Equipment.location_id`, when set, must belong to
     `Equipment.client_id`") are enforced and tested with a failing case.
   - Use `factory-boy` factories. Mark integration tests with
     `@pytest.mark.integration`.

## Hard rules

- **No floats for money.** `decimal.Decimal` end-to-end, persisted as
  `NUMERIC(12, 2)` via `service_crm.shared.money.Money`.
- **No `datetime.now()` in models.** Use `service_crm.shared.clock.now`.
- **No raw UUID PKs.** Use the `ULID` type from `service_crm.shared.ulid`.
- **Soft-delete preferred.** If an entity has financial or service
  history, add an `is_active` flag, not a hard `DELETE`.
- **Audit by inheritance.** Inherit `Auditable` so `after_*` listeners
  produce audit events automatically. Don't call an `audit(...)` helper
  by hand. *Exception: `TicketStatusHistory` is the audit; it does not
  inherit `Auditable`.*
- **No business logic in `models.py`.** Validation that needs the DB
  goes into `services.py`. Models hold structure, relationships, and
  cheap computed properties.
- **No raw SQL** in `routes.py` or templates. Ever.
- **Migrations must be reversible** unless the user explicitly approves
  a forward-only migration (and the downgrade docstring says why).
- **DB stores stable English codes** for enums and lookup tables
  (`TicketStatus`, `TicketType.code`, `TicketPriority.code`,
  `EquipmentControllerType.code`, `ProcedureTag.code`). UI translates
  display labels via Flask-Babel; codes never reach the user.
- **Lookup tables ship with a seed.** Adding `TicketType` etc. requires
  a data migration that inserts a project-default set of rows
  (e.g. `incident`, `preventive`, `commissioning`, `warranty`).

## Stop conditions

- The model needs a relationship that wasn't in the agreed sketch.
- A migration would be destructive on production data.
- A new column would silently break an existing query.
- The change requires a new top-level dependency.

## Definition of done

- `flask --app service_crm db upgrade` and `db downgrade -1` round-trip
  cleanly on SQLite.
- `pytest tests/<bp>/` passes with the new tests.
- `ruff check`, `ruff format --check`, and `mypy service_crm` all pass.
- The migration file's docstring summarises the change in one sentence.
